## #76 — Wire Sentry into /api/v1 catch blocks

**Branch:** `feature/76-sentry-api-v1` → `main`
**Files:** 6 changed (+233 / −3)
**Tests:** 358/358 passing

### Why
We had `@sentry/nextjs` installed but the v1 API surface used raw `console.error` for catch blocks. That worked locally and was invisible in production. This PR makes the API surface report exceptions through one helper so we can stop guessing which routes are failing.

### What
- New `lib/utils/api-sentry.ts` exporting `reportApiError(err, ctx: ApiErrorContext)` where `ApiErrorContext = { route, method, requestId?, workspaceId?, userId?, extra? }`.
- The helper wraps `Sentry.withScope` to set tags `surface=api/v1`, `route`, `method`, `request_id`, `workspace_id`, `user_id`, then calls `captureException`. Wrapped in its own try/catch so observability never breaks the request path.
- In dev mode it also writes `console.error` so local debugging is unchanged.
- Replaces all `console.error` calls in `app/api/v1/**/route.ts` catch blocks with `reportApiError` calls.
- Sentry SDK only initialises if `SENTRY_DSN` is set — local dev stays quiet.

### How to verify
1. Set `SENTRY_DSN` in `.env.local`.
2. Trigger any v1 route with a deliberate failure (e.g. malformed workspace id).
3. Confirm the event lands in Sentry with the expected tags.

### Risks
- One helper, one grep target. Future route authors must use it instead of raw `Sentry.captureException` — enforced by code review only, not by lint.

## #77 — Per-device session list with revoke

**Branch:** `feature/77-per-device-sessions` → `main` (after #76)
**Files:** 12 changed (+853 / −13)
**Tests:** 367/367 passing

### Why
The Profile → Sessions tab was an honest empty state with the words "Coming soon." This closes the loop on roadmap item #30 — users can now see every active session with device, browser, IP, last-seen-at, and revoke any session except the current one.

### What
- New SQL migration `supabase/migrations/20260429000001_user_sessions.sql` defining two `SECURITY DEFINER` functions:
  - `public.list_user_sessions(uuid)` — returns all active sessions for a user, parsed from Supabase's auth schema.
  - `public.revoke_user_session(uuid, uuid)` — revokes one session by id, scoped to the calling user.
  - Both `SET search_path = pg_catalog, public` and are granted to `service_role` only.
- New `lib/data/sessions.ts` wrapping the two RPCs.
- `app/(app)/workspace/[slug]/profile/ProfileClient.tsx` updated: replaces the empty state with a real list including a "This device" card and "All active sessions" with Revoke buttons.

### How to verify
1. Run the migration against a Supabase project.
2. Sign in from two browsers.
3. Profile → Sessions on browser A should show two rows; revoking the other one should sign out browser B on next request.

### Risks
- The migration is the only schema change. Rollback is `DROP FUNCTION` for both names.
- The route layer checks ownership before calling the RPC; the RPC checks again. Two checks for a destructive operation is intentional.

## #79 — Auto-generate SDK types from OpenAPI

**Branch:** `feature/79-sdk-openapi-types` → `main` (after #78)
**Files:** 8 changed (+1,979 / −8)
**Tests:** 353/353 passing

### Why
The public SDK's request and response types were hand-written and drifted every time a route changed. This PR makes the types a derived artefact of the OpenAPI document, regenerated by one command.

### What
- New `scripts/generate-sdk-types.ts` (tsx script): imports `buildOpenApiSpec()` from the existing `lib/utils/openapi`, runs `openapi-typescript` against it, writes `packages/sdk/src/types.ts` with an auto-generated banner + `eslint-disable`.
- New devDeps: `openapi-typescript ^7.13.0`, `tsx`.
- New npm script: `sdk:generate-types`.
- `packages/sdk/src/types.ts` (new, ~38 KB) is checked in — consumers don't need to run code generation.
- `packages/sdk/src/index.ts` re-exports `paths`, `components`, `operations`.

### How to verify
1. `npm run sdk:generate-types`
2. `git diff packages/sdk/src/types.ts` should be clean unless the API surface actually changed.
3. CI can `npm run sdk:generate-types && git diff --exit-code packages/sdk/src/types.ts` to fail PRs that drift.

### Risks
- The script imports `buildOpenApiSpec` directly — it does NOT hit the running server. So no network dependency in CI.
- Build warning about a transitive `web-worker` dependency from Scalar (lands later in #80) is unrelated.

## #81 — GitHub OAuth adapter (#15/#31 reference adapter)

**Branch:** `feature/81-oauth-github-adapter` → `main` (after #76)
**Files:** 10 changed (+960 / −15)
**Tests:** 370/370 passing (12 new)

### Why
The OAuth scaffolding has been in for a few releases. This is the first concrete adapter on it — chosen as the reference because GitHub has the smallest scope surface, well-documented OAuth web flow, and is useful for both Settings → Integrations (#31) and the Import Modal (#15).

### What
- New `lib/oauth/github.ts`: adapter exporting `buildAuthorizeUrl` + `exchangeCode`. `pkce: false` (GitHub web flow uses client secret). Conservative scopes: `read = read:user + user:email`; `write` adds `public_repo`.
- `app/api/v1/oauth/[provider]/start/route.ts`: replaced the 501 stub with a real authorize-redirect. Mints a CSRF state token via `randomBytes(24).toString('base64url')`, sets a 10-minute SameSite=Lax httpOnly cookie carrying `{ state, workspaceId, mode }`.
- New `app/api/v1/oauth/[provider]/callback/route.ts`: verifies state cookie, re-checks workspace membership, calls `adapter.exchangeCode`, encrypts the token envelope via `sealTokenEnvelope`, upserts to `oauth_connections` by `(workspace_id, provider, external_id)`. Errors redirect with `?oauth_error=<code>` and report to Sentry; never leak stack traces.
- New `tests/unit/oauth-github.test.ts` (12 cases) — URL building, scope mapping, env validation, fetch-mocked happy path, error responses, edge cases (null name, expires_in, missing user fields).

### How to verify
1. Register a GitHub OAuth app pointing at `<origin>/api/v1/oauth/github/callback`.
2. Set `LAZYNEXT_OAUTH_GITHUB_CLIENT_ID` and `LAZYNEXT_OAUTH_GITHUB_CLIENT_SECRET`.
3. Settings → Integrations → Connect on the GitHub tile.
4. Confirm a row appears in `oauth_connections` with the encrypted envelope.

### Risks
- Without env vars, `isProviderConfigured('github')` returns false and the start route 503s with the env names to set. UI hides the Connect button. Safe.

## #85 — Linear OAuth adapter

**Branch:** `feature/85-oauth-linear-adapter` → `main` (after #84)
**Files:** 14 changed (+1,676 / −15) cumulative, 4 new on top of #84
**Tests:** 391/391 passing (11 new)

### What (delta vs #84)
- New `lib/oauth/linear.ts`: standard OAuth2. `pkce: false`. **Comma-separated** scopes (not space-separated like GitHub). Token endpoint takes form-encoded creds in the body. Identity comes from a separate GraphQL `viewer` query — not inline like Notion's token response. Scope strategy: `read=read`, `write=read,write`, `admin=read,write,admin`. We avoid the granular `issues:create` / `comments:create` scopes because the realistic Lazynext use case (mirror issues, link decisions) inherently needs read.
- Start + callback routes register the Linear adapter.
- New `tests/unit/oauth-linear.test.ts` (11 cases).

### How to verify
1. Create a Linear OAuth application.
2. Set `LAZYNEXT_OAUTH_LINEAR_CLIENT_ID` and `LAZYNEXT_OAUTH_LINEAR_CLIENT_SECRET`.
3. Connect from Settings → Integrations → Linear.

### Risks
- The Linear `prompt=consent` parameter forces a fresh consent screen even if the user has previously authorised. We want every scope upgrade to be visible.

## #86 — Slack OAuth adapter

**Branch:** `feature/86-oauth-slack-adapter` → `main` (after #85)
**Files:** 16 changed (+1,993 / −15) cumulative, 4 new on top of #85
**Tests:** 400/400 passing (9 new)

### What (delta vs #85)
- New `lib/oauth/slack.ts`: Slack OAuth v2 (`oauth.v2.access`). Three quirks the adapter handles: (1) Slack returns HTTP 200 with `{ ok: false, error: '...' }` on failures — the `ok` field is the only reliable signal. (2) Two scope axes — we send empty `user_scope` to avoid "which user is this acting as" surprises and store the bot token (`xoxb-`) as the canonical credential. (3) Refresh tokens are not issued unless token rotation is enabled at the app level; we leave it off.
- `external_id` is set to `team.id`, not the bot user id — re-installing into the same Slack workspace upserts on `(lazynext_workspace_id, 'slack', team.id)`.
- Scope strategy: `read=channels:read+team:read`, `write` adds `chat:write+chat:write.public`, `admin` adds `channels:manage+groups:write`.
- Start + callback routes register the Slack adapter.
- New `tests/unit/oauth-slack.test.ts` (9 cases).

### How to verify
1. Create a Slack app, OAuth & Permissions → Redirect URLs `<origin>/api/v1/oauth/slack/callback`. Bot Token Scopes match the `read`/`write` lists above.
2. Set `LAZYNEXT_OAUTH_SLACK_CLIENT_ID` and `LAZYNEXT_OAUTH_SLACK_CLIENT_SECRET`.
3. Connect from Settings → Integrations → Slack.

### Risks
- `chat:write.public` lets the bot post in channels it hasn't been invited to. Default to write-mode for the typical Lazynext use case (post decision summaries) but document this in the consent prompt copy.
