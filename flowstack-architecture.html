01 — Foundation
FlowStack Overview
A unified spatial canvas platform that replaces Asana, Notion, Airtable, Slack (contextual), and Zapier — built for zero infrastructure cost at launch, scalable to 1M+ users without a rewrite.

Platform Philosophy

FlowStack is built on a single architectural insight: every workflow tool in existence is just one of seven fundamental primitives dressed up with a brand name. By unifying these primitives into a shared graph — where every object can reference, trigger, or embed every other object — FlowStack eliminates the coordination tax teams pay switching between disconnected tools.

Design Principles
Graph-firstEvery primitive is a node. Every relationship is an edge. The entire platform is a queryable graph.
Real-time nativeMultiplayer collaboration is not a feature — it is the default state of every object.
AI-readableThe workflow graph is always serializable to JSON and passable directly to Claude for analysis.
Zero infrastructure ownershipEvery backend service is managed. No servers, no DevOps, no ops team required to launch.
JSONB flexibilityEach node stores type-specific data in a JSONB column. New node types require zero schema migrations.

India-Safe Stack Constraint

The architecture explicitly avoids Supabase (blocked in India under IT Act Section 69A) and uses Neon.tech + Clerk.dev as direct replacements with identical or superior developer experience.

04 — Architecture
System Architecture
The complete request lifecycle and service topology — from browser to database and back.

Request Lifecycle

A typical mutation request (e.g. user creates a task node) flows through each layer as follows:

1. User drags a TASK node onto canvas (React Flow)
2. Zustand optimistic update → canvas renders immediately
3. POST /api/nodes → Next.js API Route
4. Clerk middleware validates JWT → extracts workspace_id
5. Drizzle INSERT into nodes table with workspace_id guard
6. Inngest event dispatched: "node.created"
7. Inngest evaluates automation triggers for this workspace
8. If automation matches → executes action (creates edge, notifies, etc.)
9. Liveblocks broadcasts canvas delta to all connected users
10. Other users' canvases update in real-time without refresh

Workspace Scoping Pattern

Since we're not using Supabase Row Level Security, workspace isolation is enforced by a shared query helper that wraps every database operation:

// lib/db/scoped-query.ts
import { auth } from '@clerk/nextjs/server'
import { db } from './client'
import { eq, and } from 'drizzle-orm'

// Every query in the codebase goes through this helper
// It's IMPOSSIBLE to forget workspace scoping
export async function scopedQuery() {
  const { userId, orgId } = auth()
  if (!userId || !orgId) throw new Error('Unauthorized')

  return {
    workspaceId: orgId,

    // Usage: const { findNodes } = await scopedQuery()
    // findNodes() ALWAYS includes workspace_id filter
    findNodes: (workflowId: string) =>
      db.select().from(nodes).where(
        and(
          eq(nodes.workspaceId, orgId),  // ← always present
          eq(nodes.workflowId, workflowId)
        )
      ),

    createNode: (data: NewNode) =>
      db.insert(nodes).values({
        ...data,
        workspaceId: orgId,  // ← injected, not caller-supplied
      }).returning()
  }
}

07 — Architecture
Real-time & Sync
How multiple users collaborate on a canvas simultaneously without conflicts or stale state.

Two-layer Real-time Strategy

Layer 1 — Presence (Liveblocks) Managed

Handles ephemeral, high-frequency events: cursor positions, hover states, active selections, typing indicators. These don't touch the database — they're pure presence signals.

Layer 2 — Data Sync (Optimistic + Server Reconciliation)

Mutations flow as: optimistic local update → API write to Neon → broadcast to other connected clients. Last-write-wins on position. Operational transforms (via Liveblocks) for collaborative text editing inside Doc nodes.

Canvas Sync Flow

// Optimistic node move — user drags a node
function onNodeDragStop(event, node) {
  // 1. Update Zustand immediately (instant render)
  updateNode(node.id, { position: node.position })

  // 2. Debounced DB write (500ms) to avoid hammering API
  debouncedSave(node.id, {
    position_x: node.position.x,
    position_y: node.position.y
  })

  // 3. Liveblocks broadcasts position to co-editors via presence
  updatePresence({ dragging: { nodeId: node.id, pos: node.position } })
}

Conflict Resolution Strategy

Content TypeStrategyRationale
Node position (x, y)Last-write-winsNo meaningful merge possible. Last user to move wins.
Node status, priorityLast-write-wins with timestampScalar field, latest timestamp wins
Doc content (rich text)OT via LiveblocksGoogle Docs-style — character-level merging
Table cell contentLast-write-wins per cellCell-level granularity avoids row-level conflicts

08 — Architecture
AI Layer — FlowMind
How Claude reads, analyzes, and generates workflow graphs. No fine-tuning required — the graph serializes directly into context.

Core Architecture Principle

FlowMind doesn't need a vector database, fine-tuning, or embeddings for primary features. The entire workflow graph serializes to a compact JSON representation that fits in Claude's context window. This makes it trivially simple to build AI features — serialize the graph, pass it to Claude with a task-specific system prompt, parse the response.

FlowMind API Route

// app/api/ai/analyze/route.ts
export async function POST(req: Request) {
  const { workflowId, question } = await req.json()
  const { workspaceId, findWorkflow } = await scopedQuery()

  // Serialize entire graph to compact JSON
  const workflow = await findWorkflow(workflowId)
  const graphContext = serializeForAI(workflow)

  const response = await fetch('https://api.anthropic.com/v1/messages', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({
      model: 'claude-sonnet-4-20250514',
      max_tokens: 1000,
      system: `You are FlowMind, an AI assistant that reads and analyzes
workflow graphs for teams. You understand 7 node types: task, doc,
table, thread, decision, automation, and pulse. You identify
bottlenecks (nodes with no downstream connections and status "stuck"),
overloaded assignees (users with >5 open tasks), and stalled paths
(nodes unchanged >7 days with incomplete status). Always cite specific
node titles and assignees. Be direct and actionable.`,
      messages: [{
        role: 'user',
        content: `Workflow graph:\n${graphContext}\n\nQuestion: ${question}`
      }]
    })
  })

  const data = await response.json()
  return Response.json({
    answer: data.content[0].text
  })
}

Automations Engine
How AUTOMATION nodes execute — from trigger detection to action execution — powered by Inngest.

Automation Execution Pipeline

// Step 1: Database webhook fires when any node is updated
// (configured in Neon via pg_notify or Trigger.dev)

// Step 2: Inngest function receives event
export const processAutomation = inngest.createFunction(
  { id: 'process-automation-trigger' },
  { event: 'flowstack/node.updated' },

  async ({ event, step }) => {
    const { nodeId, workspaceId, changes } = event.data

    // Step 3: Find all automations in this workspace
    const automations = await step.run('fetch-automations', async () =>
      getAutomationNodes(workspaceId)
    )

    // Step 4: Evaluate each automation's trigger condition
    for (const automation of automations) {
      const trigger = automation.data.trigger
      if (matchesTrigger(trigger, changes)) {
        // Step 5: Execute each action
        await step.run(`execute-${automation.id}`, async () =>
          executeActions(automation.data.actions, workspaceId)
        )
      }
    }
  }
)

// matchesTrigger — pure function, easy to test
function matchesTrigger(trigger: TriggerConfig, changes: NodeChanges) {
  if (trigger.type === 'node_status_change') {
    return changes.status?.new === trigger.condition.status
  }
  if (trigger.type === 'due_date_passed') {
    return changes.due_at?.new < new Date()
  }
  return false
}

export const workflows = pgTable('workflows', {
  id:           uuid('id').primaryKey().defaultRandom(),
  workspaceId:  uuid('workspace_id').notNull().references(() => workspaces.id),
  name:         text('name').notNull(),
  description:  text('description'),
  canvasState:  jsonb('canvas_state'),    -- viewport position, zoom
  isTemplate:   boolean('is_template').default(false),
  isPublic:     boolean('is_public').default(false),
  createdBy:    text('created_by').notNull(),  -- Clerk user ID
  createdAt:    timestamp('created_at').defaultNow(),
  updatedAt:    timestamp('updated_at').defaultNow(),
})

export const nodes = pgTable('nodes', {
  id:          uuid('id').primaryKey().defaultRandom(),
  workspaceId: uuid('workspace_id').notNull().references(() => workspaces.id),
  workflowId:  uuid('workflow_id').notNull().references(() => workflows.id),
  -- Node type discriminator
  type:        text('type').notNull(),    -- task|doc|table|thread|decision|automation|pulse

  positionX:   integer('position_x').notNull().default(0),

  -- Type-specific data — no schema migrations for new node types
  data:        jsonb('data').notNull().default({}),
  -- Shared fields surfaced for performance (avoid JSONB scan)
  status:      text('status'),
  assignedTo:  text('assigned_to'),
  dueAt:       timestamp('due_at'),
  createdBy:   text('created_by').notNull(),
  createdAt:   timestamp('created_at').defaultNow(),
  updatedAt:   timestamp('updated_at').defaultNow(),
})

Multi-tenancy Model
How complete workspace isolation is enforced without Supabase RLS — using application-level guards and Clerk organizations.

Isolation Architecture

Shared Database, Logical Isolation Recommended for solo founder

All workspaces share one Neon Postgres instance. Isolation is enforced by including workspace_id in every query — injected automatically by the scopedQuery() helper, never supplied by the API caller. This is the right model until ~$5M ARR when enterprise compliance requirements may demand schema-per-tenant.

Clerk Organizations → Workspace Mapping

// Clerk org ID maps 1:1 to FlowStack workspace
// When a user signs up, middleware creates workspace automatically

export async function middleware(req: NextRequest) {
  const { userId, orgId } = getAuth(req)

  if (userId && !orgId) {
    // User exists but has no org → redirect to workspace creation
    return NextResponse.redirect('/onboarding/create-workspace')
  }

  if (userId && orgId) {
    // Ensure workspace record exists in DB
    await ensureWorkspaceExists(orgId, userId)
  }
}

13 — Operations
Security Model
Defense in depth — four layers of security from network edge to database row.
Layer 1 — Network Edge (Vercel + Cloudflare)
  • DDoS protection via Cloudflare in front of Vercel
  • Automatic HTTPS — no HTTP allowed
  • Rate limiting on all API routes via Upstash Redis (100 req/min per user)
  • Bot detection on auth endpoints
Layer 2 — Authentication (Clerk.dev)
  • JWT-based sessions with automatic rotation
  • MFA available on all plans, enforced on Enterprise
  • SSO (SAML/OIDC) for Enterprise tier
  • Session revocation on password change or suspicious activity
Layer 3 — Authorization (Application Layer)
  • workspace_id injected into every query via scopedQuery() helper
  • Role-based access: owner, admin, member, guest
  • Guest access: read-only, scoped to specific workflows only
  • API keys for external integrations — scoped per workspace
Layer 4 — Data (Neon Postgres)
  • Encryption at rest (AES-256) — Neon default
  • Encryption in transit (TLS 1.3) on all DB connections
  • Point-in-time recovery (PITR) — 7 days on free, 30 days on Pro
  • Connection pooling via Neon serverless driver
